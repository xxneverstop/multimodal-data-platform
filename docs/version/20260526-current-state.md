# 2026-05-26 当前前后端详细情况

## 1. 文档目的

这份文档用于记录 2026-05-26 时点 MMDP 的当前真实状态，重点说明：

- 当前平台对象主线是什么
- 前端页面现在如何组织
- 后端接口和对象语义现在是什么
- 哪些能力已经真实可用
- 哪些能力仍然属于 MVP 或占位实现

本文档不描述理想态，不引入新的业务设计，只描述当前仓库中的前后端实现现状。

## 2. 当前统一对象主线

当前系统更适合统一理解为一条主线：

`Task -> Asset -> Processing Job -> Derived Asset -> Lineage`

其中：

- `Task` 是业务任务容器
- `Asset` 是平台内的核心数据对象
- `Processing Job` 是一次处理记录
- `Derived Asset` 是处理后的派生产物
- `Lineage` 用于追踪任务、资产、处理作业之间的关系

当前还存在两条明确的数据接入分支：

### 2.1 平台上传分支

通过文件上传进入平台后，系统会：

- 写入 `data_file`
- 自动生成 `UPLOADED_FILE` 来源的 `data_asset`
- 自动执行当前 MVP 级 QC
- 写入 `qc_report`

### 2.2 外部登记分支

通过外部路径登记进入平台后，系统会：

- 只写资产元数据和外部路径
- 不上传文件
- 不写 `data_file`
- 不自动执行 QC

## 3. 当前前端情况

### 3.1 前端技术与范围

当前前端位于 `mmdp-frontend/`，技术栈为：

- Vue 3
- Vite
- TypeScript

本轮前端已经完成一次较大幅度的信息架构重构，重点集中在任务详情页表达层，不改后端接口语义，不改 API 调用方式。

### 3.2 当前页面路由现状

当前路由已转向“平台化”组织方式，核心页面包括：

- 平台概览
- 数据资产
- 资产接入
- 采集任务
- Session
- 处理
- 标注
- 质检
- 成品资产
- 管理模块

旧的 `/tasks` 路由已保留重定向能力，但主入口已经转向 `/acquisition` 体系。

### 3.3 任务详情页当前信息架构

任务详情页已从“一滑到底工作台”重构为按阶段分组的结构，顶部保留任务上下文，主体改为一级 Tabs。

当前一级 Tabs 为：

- 任务概览
- 资产管理
- 处理流程
- 质量检查
- 数据链路

页面不再把上传、处理条件、处理作业、质量检查、数据链路混在一个长页面里，而是围绕对象生命周期拆分展示。

### 3.4 任务概览页当前语义

任务概览当前只承担“任务容器说明”职责，主要展示：

- 任务基础信息
- 当前数据状态
- 任务与资产、处理作业、派生资产、链路之间的关系摘要
- 两条数据接入分支说明

这里不再承载具体操作入口，避免把任务对象和资产处理动作混为一谈。

### 3.5 资产管理页当前语义

资产管理已经成为任务详情页的主入口，内部按两种接入方式分组：

- 平台上传
- 外部登记

#### 平台上传

页面文案明确表达：

- 这是“数据接入”
- 通过上传文件进入平台
- 上传后自动写 `data_file`
- 自动生成 `UPLOADED_FILE` 资产
- 自动执行当前 MVP 级 QC

#### 外部登记

页面文案明确表达：

- 这是“资产登记”
- 只登记外部路径和资产元数据
- 不上传文件
- 不写 `data_file`
- 不自动执行 QC

#### 资产列表

资产列表已增加“来源”表达，用于强化资产对象模型。目前前端展示会区分：

- 平台上传
- 外部路径
- 处理生成

来源展示本质上是从现有字段推导而来，不改后端接口：

- `producedByJobId` 有值时，视为“处理生成”
- `sourceType === UPLOADED_FILE` 时，视为“平台上传”
- `sourceType === EXTERNAL_PATH` 时，视为“外部路径”

这有助于用户理解：

- 资产生命周期不同
- 是否有关联文件不同
- 是否自动触发 QC 不同
- 是否由处理流程派生不同

### 3.6 处理流程页当前语义

处理流程页已经不再把 `Pipeline` 当一级对象强调，而是围绕 `Processing Job` 组织。

当前页面结构是：

- 处理条件检查
- 自动处理
- 人工登记

#### 处理条件检查

这里仍然复用后端 `fetchAvailablePipelines` 返回的数据，但前端语义已经弱化 `Pipeline` 概念。

当前表达重点是：

- 当前可执行处理
- 所需资产类型
- 已有资产类型
- 缺失资产类型
- 当前状态原值 `READY / MISSING_REQUIRED_ASSETS`

页面明确提示：

“当前检查只判断资产类型是否齐全，不代表算法已经真实跑通。”

#### 自动处理

自动处理是未来平台主线方向，但当前仍属于占位执行能力。

前端当前已明确提示：

- 自动处理代表未来平台主线能力
- 当前执行仍是占位执行
- 返回结果不是真实算法结果

#### 人工登记

人工登记仍保留，但已经被弱化为次级区域，用于记录线下处理过程，而不是表现为平台核心自动能力。

页面语义是：

- 用于把线下处理过程纳入平台追踪
- 会登记输入资产、输出资产和处理记录
- 会写入后续可查看的数据链路

### 3.7 质量检查页当前语义

质量检查页只展示上传文件触发的 `qc_report`。

前端当前已经明确表达：

- 当前 QC 仅对平台上传文件自动执行
- 外部资产不会自动生成 QC 报告

也就是说，当前页面展示的是“上传文件检查结果”，不是“任务整体质量结论”，也不是“全部资产质量结论”。

### 3.8 数据链路页当前语义

数据链路页放在任务详情页最后一个 Tab，只作为结果视图使用。

当前职责是：

- 展示 task / asset / processing job 之间的关系
- 作为追踪视图，而不是主要操作入口

页面定位已经从“操作入口”调整为“结果查看”。

### 3.9 前端文案当前状态

当前前端已在主要页面中推进中文化，目标是：

- 尽量去掉普通英文
- 保留必要技术枚举值

当前保留英文原值的范围主要是技术枚举值，例如：

- `RGB_MOCAP_ALIGNMENT`
- `MOCAP_CSV`
- `SMPL_RESULT`
- `UPLOADED_FILE`
- `EXTERNAL_PATH`
- `MOCK`
- `MANUAL`

普通界面术语已经尽量统一为：

- 数据接入
- 资产登记
- 处理作业
- 派生资产
- 质量检查
- 数据链路

## 4. 当前后端情况

### 4.1 后端技术与职责

当前后端位于 `mmdp-backend/`，技术栈为：

- Spring Boot
- MyBatis-Plus
- OSS

后端当前已经形成任务、文件、资产、质检、处理、链路这几条主能力线。

### 4.2 当前后端核心模块

当前主要模块包括：

- `task`
- `file`
- `asset`
- `qc`
- `processing`

对应的接口控制器已经存在：

- `AcquisitionTaskController`
- `DataFileController`
- `DataAssetController`
- `QcReportController`
- `PipelineController`
- `ProcessingJobController`

### 4.3 Task 当前语义

`Task` 当前是业务任务容器，后端会保存其基础元数据，例如：

- 任务名称
- 被试编号
- 动作名
- 采集日期
- 场景和操作员等上下文

当前 `TaskStatus` 包括：

- `CREATED`
- `UPLOADED`
- `QC_PASSED`
- `QC_WARNING`
- `QC_FAILED`

但需要注意：

这些状态当前混合了承载“任务生命周期”和“最近一次上传检查结果”的意味，严格来说并不等于任务整体最终状态。

前端当前已经开始弱化这种误导性表达，但后端状态模型本身尚未拆分。

### 4.4 文件上传链路当前语义

文件上传是当前后端中最完整的一条真实链路。

当前流程是：

1. 校验任务存在
2. 校验上传文件非空
3. 上传文件到 OSS
4. 将任务状态先更新为 `UPLOADED`
5. 写入 `data_file`
6. 自动创建一条来源为 `UPLOADED_FILE` 的资产
7. 同步执行 QC
8. 写入 `qc_report`
9. 再按 QC 结果把任务状态改为 `QC_PASSED / QC_WARNING / QC_FAILED`

这条链路当前是真实可用能力，不是 mock。

### 4.5 资产模型当前语义

当前后端资产模型至少支持三类来源：

#### 1. 上传资产

由文件上传自动生成：

- `sourceType = UPLOADED_FILE`
- 关联 `fileId`

#### 2. 外部资产

由外部登记生成：

- `sourceType = EXTERNAL_PATH`
- 保存 `externalPath`
- 不关联 `data_file`

#### 3. 派生资产

由处理作业生成：

- 记录 `producedByJobId`
- 可以来自文件或外部路径

因此后端已经具备较清晰的资产来源模型，前端这次重构只是把这个模型表达得更清楚。

### 4.6 处理条件检查当前语义

当前后端仍保留 `available pipelines` 接口，但它的真实含义并不是“可编排 pipeline 系统”，而是：

“当前资产是否满足某类处理的最小输入条件”

当前只检查一个处理能力：

- `RGB_MOCAP_ALIGNMENT`

当前必需资产是：

- `MOCAP_CSV`
- `SMPL_RESULT`

当必需资产齐全时，状态返回：

- `READY`

否则返回：

- `MISSING_REQUIRED_ASSETS`

因此这部分当前本质上是 readiness check，不是 DAG、调度系统或真实算法编排引擎。

### 4.7 Processing Job 当前语义

当前后端已支持两类处理作业：

#### 1. 自动处理作业

调用 `POST /processing-jobs` 后：

- 校验任务存在
- 只接受 `RGB_MOCAP_ALIGNMENT`
- 检查 `MOCAP_CSV` 和 `SMPL_RESULT` 是否齐全
- 创建 `ProcessingJob`
- `executorType` 写为 `MOCK`
- 交给占位执行器执行

这条能力当前的“作业记录”是真实的，但“算法执行结果”仍是 mock。

#### 2. 人工登记作业

调用 `POST /processing-jobs/manual` 后：

- 校验输入资产属于当前任务
- 创建一条 `executorType = MANUAL` 的处理作业
- 根据请求生成输出资产
- 为输入资产和输出资产写入 lineage 关系

这条能力的意义不是自动算法执行，而是把线下处理过程纳入平台追踪体系。

### 4.8 QC 当前语义

当前 QC 是真实执行，但能力仍属于 MVP 级规则检查，不是深度业务校验。

当前实现主要会检查：

- 文件是否为空
- 扩展名是否支持
- 文件大小是否异常
- 文本、CSV、JSON 等样本内容是否可读
- 针对 `MOCAP_CSV`、`SMPL_RESULT`、`RGB_VIDEO_MP4` 等类型做一些有限规则判断

当前支持的检查更接近：

- 文件格式合理性
- 样例内容可解析性
- 少量字段和头部启发式规则

还不是：

- 完整数据质量体系
- 跨模态一致性校验
- 深度算法质量评估

### 4.9 数据链路当前语义

后端当前已经支持按任务生成链路图视图。

当前链路节点包括：

- task 节点
- asset 节点
- job 节点

当前链路边包括：

- task 到原始资产
- 输入资产到 job
- job 到输出资产

因此当前链路能力已经能够支持：

- 查看原始资产从哪里进入任务
- 查看哪些输入资产进入了哪次处理
- 查看处理后生成了哪些派生资产

## 5. 当前真实能力与占位能力分层

### 5.1 当前真实能力

当前已经真实可用的能力包括：

- 任务创建与基础信息管理
- 文件上传到 OSS
- `data_file` 持久化
- 上传后自动资产化
- 外部路径资产登记
- QC 报告生成与查询
- 处理作业记录持久化
- 人工登记处理过程
- 派生资产登记
- 输入输出资产链路记录
- 任务维度数据链路查询

### 5.2 当前 MVP 或占位能力

当前仍然属于 MVP 或占位性质的部分包括：

- `RGB_MOCAP_ALIGNMENT` 自动处理执行结果
- mock executor
- readiness check
- QC 深度不足，只是规则级检查
- 当前处理能力种类仍然非常少

因此，系统当前更适合被定义为：

“具备真实数据记录能力和追踪能力，但自动算法处理能力仍处于占位阶段的实验室内部数据平台 MVP”

## 6. 当前前后端边界

### 6.1 前端当前负责

- 围绕任务组织平台阶段
- 将资产接入、处理记录、质量检查、数据链路按生命周期分组
- 把后端已有对象语义表达清楚
- 避免把 readiness check 误导为真实 pipeline

### 6.2 后端当前负责

- 真实保存任务、文件、资产、QC、处理作业和链路数据
- 提供上传、外部登记、自动处理、人工登记、链路查询等 API
- 在不引入复杂编排引擎的前提下，提供最小闭环

## 7. 当前系统最准确的阶段性定义

截至 2026-05-26，MMDP 当前最准确的定义是：

### 7.1 平台主线

系统已经形成以下可追踪主线：

`Task -> Asset -> Processing Job -> Derived Asset -> Lineage`

### 7.2 数据接入方式

系统当前支持两种接入方式：

- 平台上传
- 外部登记

### 7.3 自动处理能力现状

系统当前并不应被表述为已经具备完整 pipeline 编排能力。

更准确的说法是：

- 系统具备“处理条件检查”
- 系统具备“处理作业记录”
- 系统具备“派生资产登记”
- 系统具备“数据链路追踪”
- 系统的自动处理执行结果仍是 mock
