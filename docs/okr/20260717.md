## 完成事项

1. **完成了 G1 数据处理双路径输出一致性全面对比**。将同一份源数据（`episode_0.hdf5` 346 帧 + `episode_0.svo2`，MD5 完全一致）分别经平台 Worker Pipeline（G1_MERGE_CAMERA_ROBOT → G1_CONVERT_TO_LEROBOT）和独立脚本（`merge_data.py` → `convert_to_lerobot.py`）处理后，逐列对比了两个 Parquet 产物。排查过程中先定位了初始对比（511 帧 vs 346 帧）的根因——脚本路径和 Worker 路径使用了不同批次的采集数据，并非同一份源文件。切换到相同源数据后，确认两个产物在帧数（346）、列结构（11 列）、action 标签、状态向量、时间戳等所有数值列上**逐帧完全一致**

2. **完成了图像差异的根因定位与量化分析**。两个产物的 `observation.images.left` 图像在像素层面存在微小差异，但统计特征确认它们是同一帧画面：均值均为 ~125.3、标准差均为 ~48.6、绝对差中位数仅 **1**（255 色阶中）、83% 的像素差值 < 5、100% 的像素差值 < 50。差异源于两条路径各自 Python 环境中的 OpenCV 版本不同，导致 JPEG 编码器（相同 quality=95 参数）产出的字节流有微小偏差——JPEG 是有损压缩，不同版本的 DCT 系数舍入策略会产生不同但视觉等价的结果。同时识别了 LeRobot 库版本差异导致的元数据格式漂移（`_type: "List"` vs `"Sequence"`），属于 v1→v2 的格式变更，语义等价

3. **完成了对 VLA 训练影响的评估**。结论是**没有影响**：action 标签作为监督信号完全一致；图像级差（中位 1/255）远小于训练数据增强（随机裁剪、色彩抖动等通常 10-50 色阶的扰动）；元数据字段名差异仅影响加载代码的兼容性而非数据质量。同时明确了**不需要追求字节级完全一致**——那需要锁定 OpenCV、LeRobot、NumPy 及底层 C 库的精确版本，冻结整个依赖树的代价远大于收益，且对训练没有任何实际意义

4. **撰写了输出一致性分析文档**（`docs/processing/output-consistency-analysis.md`，约 2200 字），系统记录了对比方法、差异明细、根因分析、VLA 影响评估、完全一致性的代价分析，以及平台 Pipeline 完成度判定（结论：已验证可投入生产使用）

   - 在 `PipelineDefinitionServiceImpl.getAvailablePipelines()` 中增加 Worker 注册表交叉过滤，前端"可用处理"列表自动只显示 Worker 端真实可用的 Pipeline
