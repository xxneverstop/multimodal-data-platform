# MMDP 数据采集与数据平台流程说明

本文基于当前两个仓库的实现现状整理：

- `D:\workspace\multimodal-data-collector`
- `D:\workspace\multimodal-data-platform`

本文仅说明两个部分：

- 数据采集
- 数据平台

为便于说明，下文以一次“`双目摄像头 + IMU`”采集为例，描述数据如何从采集现场进入平台，并在平台中完成接收、存储和管理。本文重点说明主流程和关键对象，不展开具体字段设计，也不写成完整接口说明书。

---

## 一、数据采集

### 1.1 目标

数据采集侧的目标，是完成一次采集 Session 的发起、执行、确认、整理和上传准备，并将采集结果组织为一个具备业务语义的上传单元。

### 1.2 流程说明

一次采集通常从采集端本地页面开始。采集人员首先确认本次采集对应的业务上下文，例如所属任务、被试对象、动作内容以及采集配置，然后发起一轮新的采集 Session。对采集侧而言，核心管理单元不是单个文件，而是一次完整的采集过程。

在采集执行阶段，以双目摄像头和 IMU 为例，采集端需要同时接收视频数据和传感器数据，并在本地完成预览、实时状态展示和 Session 生命周期控制。采集侧首先要保证的是采集过程可观测、可中断、可确认，也就是说，现场人员能够及时判断当前采集是否正常、是否需要停止，以及是否需要重新开始。

当一次采集结束后，采集端会对本轮 Session 进行整理和固化，将采集结果组织为一份可归档、可上传的数据单元。该数据单元不仅包含原始文件，还包含描述本次采集身份和结构关系的元信息，用于后续平台识别这份数据属于哪个任务、对应哪次采集，以及应当如何完成入库和管理。这样形成的结果不是若干离散文件，而是一份具备业务语义的 Session 结果。

在上传阶段，采集端将本次 Session 作为一个整体提交给平台，而不是按单文件逐个进入系统。这样可以保证平台接收到的是一份上下文完整、结构完整的采集单元，便于后续统一完成任务关联、Session 建档、文件存储和结构化入库。

### 1.3 关键接口

当前采集主流程涉及的关键接口如下：

| 接口 | 作用 |
|------|------|
| `POST /session/start` | 发起一次新的采集 Session |
| `POST /session/stop` | 结束当前采集 Session |
| `POST /session/save` | 保存当前 Session 结果 |
| `POST /session/upload` | 将当前 Session 上传到平台 |
| `GET /session/current` | 查询当前 Session 状态 |

除上述接口外，采集端还提供实时状态与预览通道，用于现场确认采集状态；本文不对其交互细节展开说明。

## 二、数据平台

### 2.1 目标

数据平台侧的目标，是以任务和 Session 为核心完成数据承接、原始文件存储、结构化入库和后续统一管理，使平台能够围绕一次次采集 Session 组织数据，而不是停留在逐文件管理层面。

### 2.2 流程说明

平台的业务起点是采集任务。任务用于定义一批采集数据的归属范围和组织边界，也是后续多个采集 Session 的上层容器。因此，平台首先管理的是任务，而不是文件。

当采集端上传一次 Session 后，平台接收该次采集的归档包和元信息，并据此完成任务关联和 Session 建档。对平台而言，这一步的重点不只是“接收文件”，更重要的是识别这份数据在业务上的位置，例如它属于哪个任务、对应哪次采集、关联哪个被试和动作，以及后续应如何被查询、回放和处理。

在接收完成后，平台将原始文件写入对象存储，将任务、Session、导入记录、文件记录和资产记录等结构化元数据写入数据库。通过这一步，平台形成的不是简单的文件索引，而是一套有明确层次的数据组织关系：任务承载业务上下文，Session 承载一次采集过程，文件和资产承载具体数据实体。

从用户视角看，平台的主视角应当是“任务”和“采集 Session”。用户首先查看有哪些任务，再查看某个任务下已经发生了哪些采集，最后进入某次采集查看其中包含的数据和相关结果。这种组织方式更符合采集业务本身，也更有利于后续把回放、质检、处理和导出等能力统一收敛到 Session 维度。

### 2.3 关键接口

当前平台主流程涉及的关键接口如下：

| 接口 | 作用 |
|------|------|
| `POST /api/tasks` | 创建采集任务 |
| `GET /api/tasks` | 查询任务列表 |
| `GET /api/sessions` | 查询采集 Session 列表 |
| `POST /api/session-imports` | 接收采集端上传的 Session |
| `GET /api/tasks/{taskId}/sessions` | 查询某任务下的 Session |
| `GET /api/sessions/{sessionId}` | 查询单次 Session 详情 |

### 2.4 关键对象

当前平台主流程主要围绕以下对象组织：

| 对象 | 作用 |
|------|------|
| `acquisition_task` | 承载采集任务及其业务上下文 |
| `collection_session` | 承载一次完整采集 Session |
| `session_import_record` | 承载 Session 导入过程信息 |
| `data_file` | 承载原始文件记录 |
| `data_asset` | 承载平台视角下的数据资产记录 |

### 2.5 当前状态

从当前项目状态看，这条链路已经具备基本的数据闭环能力，即平台能够以任务为业务容器，以 Session 为采集单元，接收采集端上传的数据，并完成基础的对象存储和结构化入库。不过，采集侧和平台侧整体仍处于持续完善阶段，质检、处理、导出等能力目前仍属于 MVP 范围。本文描述的是当前项目中“数据采集如何形成一次 Session，以及该 Session 如何进入数据平台并被组织管理”的现状流程。
