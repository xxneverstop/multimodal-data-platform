# 概述

具身智能数据采集平台：用于采集具身智能体在任务执行过程中产生的多模态数据（图像、语音、文本、传感器数据、动作数据等），包括智能体本身和外部环境，并提供数据预处理、标注、存储、管理、检索等功能的软件系统。

# 系统总体目标与设计思路

从整体定位上看，本系统并不是一个简单的数据管理或标注工具，而是一个覆盖“数据产生—数据加工—数据消费”全过程的数据生产系统。系统希望实现以下目标：

- 一是支持多设备、多模态、多场景的数据统一采集；
- 二是通过标准化处理与管理流程，保证数据质量与一致性；
- 三是形成完整的数据闭环，使每一条数据都具备可追溯、可复现和可训练的属性。

# 功能需求规格说明

- 系统模块划分

在系统结构设计上，将整体功能划分为12个模块，并根据其对系统核心能力的影响程度划分为P0、P1和P2三个优先级层级。其中，P0模块主要支撑数据的产生与可用性，P1模块提升数据质量与使用效率，P2模块更多关注系统运行安全性与工程完善度。

| -------------- | ----------------------------------------------------------------------- | ------- |
| 多模态数据采集与设备管理模块 | 负责硬件接入与数据采集任务执行。                                                        | P0      |

| 数据标注模块         | 提供多类型数据的人工与半自动标注能力，生成结构化、高质量标注数据。                                       | P0      |
| 标注审核与反馈模块      | 支持标注结果的审核、退回修正与反馈闭环，确保标注规范性与准确性。                                        | P0      |
| 数据管理模块         | 实现数据的分类、检索、版本管理，提升数据可查性与可控性。                                            | P1      |
| 字典管理模块         | 构建和维护一套跨语言、可复用的机器人行为指令词汇库。                                              | P1      |

| 任务与项目协同管理模块    | 支持采集、标注、审核等任务的分配、进度跟踪与多人协同管理。                                           | P1      |
| 用户与权限控制模块      | 基于角色的权限体系，控制不同用户对数据和功能的访问与操作范围。                                         | P2      |
| 数据导出与交付模块      | 支持按需导出指定格式与版本的数据集，满足模型训练与成果交付需求。                                        | P0      |

- 权限控制

| **角色**                | **职责**                                 | **权限**                    |
| --------------------- | -------------------------------------- | ------------------------- |
| 管理员（Administrator）    | 监督平台运营与维护，包括用户、项目和系统管理，提供数据导出、分析和决策支持。 | 用户管理、项目监控与任务分配、系统设置和数据分析。 |
| 项目经理（Project Manager） | 规划和管理项目，定义目标和资源分配，协调任务分配和团队成员资源调度。     | 查看和管理项目状态、任务分配及进度跟踪。      |

| 数据标注人员（Annotator）     | 执行数据注释任务，确保注释准确，提交结果，并配合审核工作。          | 数据注释与提交，接收任务反馈并修改数据。      |
| 数据收集人员（Collector）     | 执行数据收集任务，确保数据收集任务的质量和数量。               | 佩戴硬件设备，进行数据收集和提交。         |

- 流程闭环

**采集 → 处理 → 标注 → 审核 → 展示 → 交付**

形成具身智能数据生产的完整闭环，确保数据从原始采集到模型训练使用全过程的可追溯性与高质量。

## 1. 系统功能总体说明

本系统面向**居家场景与体育运动场景**，构建一套**覆盖数据采集、处理、标注、管理与展示的具身智能数据采集系统**，支持多模态人体运动与环境数据的高质量获取，为后续的人体姿态估计、动作理解、运动分析及具身智能模型训练提供数据基础。

系统应支持**多用户、多任务、多场景并行运行**，具备良好的扩展性与数据质量控制能力。

## 2. 多模态数据采集与设备管理模块

![](https://cdn.nlark.com/yuque/0/2026/png/58677005/1768804741562-48dd3139-dfd9-4303-a465-5bf01317ff79.png)

### 2.1. 多设备接入与兼容能力

- 系统应支持接入以下类型的采集设备：

- 机器人及人体姿态感知设备：姿态传感器、关节角度传感器、动作捕捉设备

- 视觉采集设备：RGB 相机、深度相机、多摄像机阵列

- 听觉采集设备：麦克风、麦克风阵列

- 惯性与力觉设备：IMU、压力传感器、环境压力传感器

- 遥操作与穿戴式设备：穿戴式动作捕捉系统、第一视角摄像设备、手部关节姿态采集设备

- 沉浸式交互设备：头戴式虚拟现实（VR）眼镜及配套控制手柄

### 多模态数据类型实时采集

- 系统应支持以下数据类型的实时采集：

- 图像数据：RGB 图像、深度图像

- 语音数据：环境语音、指令语音

- 文本数据：用户输入的文本指令或任务描述

- 传感器数值数据：温度、湿度、压力等环境参数

- 动作与姿态数据：关节角度、关节位移、人体或机器人姿态信息

- 支持多种采集模式：

- 用户数据模式：只采集数据采集员；

- 人机遥操作模式：数据采集员通过远程遥操作方式同时采集真人和真机的模式；

- 机器自主模式：机器通过强化学习等方式进行探索，只有机器的数据，可以是真机也可以是仿真环境。

### 采集参数配置与任务定制

系统应支持对采集过程的关键参数进行灵活配置，以适应不同实验场景和采集需求。

**功能需求：**

- 支持配置以下采集参数：

- 数据采集频率

- 图像与视频分辨率

- 数据采样精度

- 数据存储格式

- 支持基于采集任务的参数模板配置，便于重复实验快速部署。

- 支持在采集任务启动前对参数配置进行校验，避免非法或冲突配置。

### 在线与离线采集模式支持

系统应支持在不同网络条件下稳定运行，保障数据采集的连续性与完整性。

**功能需求：**

- 系统应支持**在线采集模式**：

- 在网络稳定条件下，将采集数据实时上传至数据管理或处理模块

- 支持实时状态反馈与异常提示

- 系统应支持**离线采集模式**：

- 在无网络或网络不稳定环境下，将采集数据本地缓存存储

- 缓存数据应包含完整元数据与时间信息

- 系统应支持网络恢复后的**自动数据同步机制**：

- 自动检测网络状态变化

- 支持断点续传

- 避免数据重复或丢失

### 采集阶段异常检测与反馈

系统应在数据采集阶段提供基础的异常检测与反馈能力。

**功能需求：**

- 支持检测设备掉线、数据中断、异常采样等情况
- 支持采集异常的实时提示与日志记录
- 支持异常采集任务的中止或重启

## 3. 数据导入与数据上传模块

![](https://cdn.nlark.com/yuque/0/2026/png/58677005/1768804337393-100d0605-41db-430b-bd43-ad1d9f836dd6.png)

### 3.1. 多模态数据上传能力

- **多格式支持**：适配机器人与行为研究领域的主流数据格式，满足多样化的原始数据上传需求：

- 机器人数据：MCAP（推荐多模态格式）、BAG（ROS 标准格式）、LeRobot 文件夹

- 科学数据：HDF5（大容量科学数据）

- 人体动作数据：BVH（动作捕捉可视化标准）

- 多媒体数据：视频、音频、图片

- **多数据源支持**：系统应支持从以下数据来源导入数据，包括但不限于：

- 本地文件系统（手动上传的数据文件）

- 采集设备或边缘节点生成的数据包

- 外部数据采集系统或第三方平台输出的数据

- 离线采集后批量导入的数据

### 数据上传与传输管理

系统应提供稳定的数据上传机制，保障大规模数据在网络环境下的可靠传输。

**功能需求：**

- 系统应支持数据的批量上传与单文件上传。
- 系统应支持上传任务状态监控，包括上传进度、成功、失败与异常状态提示。

### 数据完整性校验与基础校验

系统应在数据导入与上传过程中对数据完整性与合法性进行基础校验。

**功能需求：**

- 系统应支持对上传数据进行完整性校验（如文件大小、校验值一致性检查）。
- 系统应支持对数据文件的基本合法性进行检查，包括格式可识别性与结构完整性。
- 对校验失败的数据，系统应提供明确的错误提示，并支持重新上传或重新导入。

### 数据元信息登记与关联

系统应在数据导入过程中自动记录与管理数据的基础元信息。

**功能需求：**

- 系统应自动记录数据导入时间、数据来源、导入方式与操作者信息。
- 系统应支持在导入时为数据关联所属任务、项目或采集批次。
- 系统应将导入的数据与后续的数据预处理模块进行关联，确保数据流转可追溯。

### 多云存储适配与管理

系统应支持使用者自行选择存储位置。

**功能需求：**

- **多厂商云存储支持**：对接国内外主流云存储服务，满足不同用户的存储偏好与合规要求：

- 国内：腾讯云 COS、阿里云 OSS

- 国际：Amazon S3、Azure Blob Storage、Cloudflare R2

- 私有部署：云存储（兼容 S3 协议 / 阿里云 OSS）

- **存储位置切换**：允许用户在不同云存储服务之间快速切换，灵活选择数据存储目的地。

## 4. 数据预处理模块

### 4.1. 数据清洗与异常处理

- 系统应支持对不同类型数据执行数据清洗操作，包括但不限于：

- 去除采集过程中的噪声数据；

- 检测并剔除异常值（如由传感器故障、通信异常导致的数据）；

- 识别并移除重复数据或冗余采样数据。

- 系统应支持针对不同数据模态（如图像数据、传感器数据、人体姿态与运动数据等）配置差异化的数据清洗策略。

- 对被清洗、剔除或标记的数据，系统应保留相应的记录与标记信息，以支持后续的数据回溯、质量分析与处理策略优化。

### 数据格式转换与统一表示

- 系统应支持解析多种设备输出的数据格式，包括但不限于：

- RAW 格式数据；

- CSV 格式数据；

- 设备自定义的日志或专用数据格式。

- 系统应支持将异构数据转换为统一的标准数据格式，包括但不限于：

- JSON；

- HDF5；

- 系统定义的结构化数据格式。

- 转换后的数据应保持原始信息的完整性，并保留必要的元数据描述信息，包括数据来源设备、时间戳、采集参数等。

- 系统应支持数据格式转换规则的可扩展配置，以适配新增设备或新增数据类型的接入需求。

### 多模态数据对齐与同步

- 系统应支持基于统一时间基准的多模态数据对齐，包括但不限于：

- 图像数据；

- 语音数据；

- 动作与人体姿态数据；

- 传感器数值数据。

- 系统应支持不同采样频率数据之间的重采样、插值或窗口化对齐处理。

- 系统应支持对数据对齐误差进行检测与评估，并对无法可靠对齐的数据进行标记或隔离处理。

- 系统应保证同一时刻或同一时间段内的多模态数据能够被准确关联与联合访问。

### 数据压缩与存储优化

- 系统应支持对图像、视频等大容量数据进行压缩处理。

- 系统应支持多种压缩方式，包括：

- 无损压缩，用于对数据完整性要求较高的应用场景；

- 有损压缩，用于对存储效率或传输效率要求较高的应用场景。

- 系统应支持基于数据类型或具体应用需求选择不同的压缩策略与参数配置。

- 系统应在数据压缩过程中保留关键元数据信息，确保压缩后的数据能够被正确解析、恢复与使用。

### 人体姿态、运动轨迹与环境特征提取

- **人体姿态特征提取**： 系统应支持从图像、深度数据或惯性/关节传感器数据中提取人体姿态信息，包括但不限于关键点位置、关节角度及其时间序列表示。
- **运动轨迹提取**： 系统应支持对人体或机器人在时间维度上的运动过程进行建模，提取位移轨迹、速度变化及运动路径等运动学特征。
- **环境特征提取**： 系统应支持从视觉、深度或环境传感器数据中提取与任务相关的环境特征，包括但不限于场景结构信息、障碍物分布及环境状态描述。
- **特征提取结果结构化输出**： 提取后的姿态、运动及环境特征应以结构化形式输出，并与原始数据保持时间与语义上的关联关系。
- **特征提取流程可配置**： 系统应支持按任务需求配置是否启用特征提取及对应的特征类型，以适配不同数据采集与应用场景。

### 处理流程可配置与可追溯性

系统应支持对数据处理流程进行配置与追溯，以适应不同实验任务与应用场景的需求。

**功能需求：**

- 系统应支持按具体任务配置数据清洗、格式转换、对齐与压缩等处理流程。
- 系统应记录每条数据在处理过程中经历的操作步骤与对应参数配置。
- 系统应支持对数据处理结果的回溯与复现，便于数据质量评估与实验结果验证。

## 5. 数据标注模块

![](https://cdn.nlark.com/yuque/0/2026/png/58677005/1768804337422-17cd61ca-2671-4380-b087-1b82e3ece671.png)

### 5.1. 多类型、多模态标注支持

- 支持多种标注类型，包括目标检测（边界框标注）、语义分割、人体姿态估计（2D/3D关键点）、动作序列标注、运动轨迹标注等；
- 支持语音转文字标注、文本指令分类等非视觉模态数据标注；
- 支持跨模态联合标注，实现图像、视频、语音、惯性数据、轨迹数据在同一时间轴上的协同标注；
- 支持单帧标注、连续时序标注与动作片段级标注，满足长序列人体运动数据需求。

### 人体姿态与动作标注能力

- 支持人体关键点与骨架结构标注，包含关节位置、连接关系及置信度信息；
- 支持人体动作与行为标签标注，如行走、抓取、弯腰、抬手等基础与复合动作；
- 支持运动轨迹标注，包括人体整体运动路径、局部肢体运动轨迹及时间区间定义；
- 支持人体与环境交互动作标注，如接触、避障、抓取对象等语义关系。

### 标注工具与协同标注机制

- 内置轻量化可视化标注工具，支持边界框绘制、关键点拖拽、轨迹绘制、时间轴操作等交互方式；
- 支持手动标注与半自动标注模式，半自动标注可基于已有模型进行预标注并人工修正；
- 支持多人同时在线标注，支持任务分配与标注进度管理，提高大规模数据标注效率；

### 标注数据关联与管理

**功能需求：**

- 确保标注信息与原始采集数据在时间戳、数据ID等维度上的精准关联；
- 支持生成带标注的数据集，用于模型训练、验证与测试；
- 标注结果以结构化形式存储（如 JSON、HDF5 等），并记录标注人、审核人、标注版本等元数据；
- 支持标注状态管理，如“未标注 / 标注中 / 待审核 / 已通过”。

## 6. 标注审核与反馈模块

### 6.1. 标注审核流程管理

- 系统应支持标注结果从“已提交”到“审核中”“审核通过”“退回修改”等完整审核流程；
- 支持按任务、项目或数据类型配置不同的审核流程与审核规则；
- 支持对审核进度与状态进行实时跟踪与管理。

### 审核内容与质量评估

- 审核人员可对标注结果的准确性、完整性与一致性进行评估；
- 支持对人体姿态标注、动作序列标注、运动轨迹标注等关键数据类型进行重点审核；
- 支持检查标注与原始数据在时间戳、空间位置等维度上的一致性；
- 支持对不符合规范的标注数据进行问题标记与说明。

### 审核反馈与修正机制

- 审核人员可对标注结果给出明确的审核意见，包括修改建议、错误类型说明等；
- 支持将审核反馈直接回传给标注人员，并形成可跟踪的修正任务；
- 支持多轮反馈与修正，直至标注结果满足审核要求；
- 审核反馈内容应与具体数据项和标注内容精确关联。

### 标注版本与变更追溯

**功能需求：**

- 系统应支持标注数据的版本管理，记录每次标注与修改的版本信息；
- 支持追溯标注结果的修改历史，包括修改人、修改时间与修改内容；
- 支持不同版本标注结果的对比与回滚，保障数据可控性；
- 审核结论应作为标注版本的重要元数据进行保存。

### 审核统计与质量分析

**功能需求：**

- 系统应支持对标注与审核过程进行统计分析，如通过率、退回率、平均审核轮次等；

# 简化后需求技术方案

## 一、总体技术实现思路

系统整体采用“边缘采集 + 中心处理 + 微服务 + 流式数据管道”的实现方式。

在实现层面，平台被拆解为多个独立部署的服务，每个服务只负责单一职责，通过 Kafka 进行数据解耦，通过 gRPC/REST 进行控制面通信，通过对象存储进行大文件交换。

## 二、核心数据工厂的具体技术实现

### 1. 设备接入与实时采集的实现方式

每类设备在采集端都由一个 **Edge Agent（边缘采集代理）** 负责。Edge Agent 采用 **C++ 或 Rust** 实现，直接运行在采集设备所在的主机或嵌入式计算单元上，避免 Python 带来的性能抖动。

设备数据通过以下方式接入：

- 相机数据通过 **RTSP + FFmpeg** 解码为帧流
- ROS2 设备通过 **DDS / rclcpp** 直接订阅 Topic
- IMU、传感器数据通过 **MQTT 或串口驱动**
- 语音数据通过 **ALSA / PortAudio**

在采集瞬间，Edge Agent 使用 **硬件时间戳 + PTP/NTP 校准**生成纳秒级时间戳，并将时间戳写入 MCAP 记录。  
所有数据先写入本地的 **环形缓冲区 + 磁盘缓存（MCAP 文件）**，在网络可用时再异步上传至平台端 Kafka 或对象存储，从而实现断点续传。

由于数据在边缘端即被序列化为 MCAP，平台端无需做二次拷贝，采集延迟可以稳定控制在 50ms 以内。

### 2. 多源数据导入的具体实现

平台端提供一个独立的 **Import Service**，该服务通过插件机制支持不同数据格式。

每一种格式都有对应的 Adapter：

- MCAP / BAG 使用 ROS2 官方解析库
- BVH 文件通过骨架解析器映射到统一关节拓扑
- HDF5 使用 h5py/C++ HDF5 API 解析 Dataset
- 视频通过 FFmpeg 提取时间戳与帧索引
- 音频通过 PCM/WAV 解码并生成帧级索引

导入过程中，系统会为每一批数据生成一个 **Asset ID**，并在 PostgreSQL 中记录数据来源、设备信息、采集参数与时间范围。  
原始文件直接存入 OSS，对象路径与 Asset ID 一一对应。

### 3. 多模态数据对齐的实现方式

多模态对齐由一个独立的 **Alignment Service** 完成，该服务基于 Apache Flink 的流式计算模型实现。

实现逻辑如下：

所有模态数据在进入 Flink 前都被转换为统一的数据结构，其中包含全局时间戳。  
Flink 以时间戳作为 Event Time，对不同模态流进行窗口化处理。在窗口内部，使用最近邻或线性插值方式进行重采样。

对于高频 IMU 数据，采用线性插值；  
对于视频与骨架数据，采用最近帧匹配。

通过这种方式，多模态数据在平台端被对齐为统一的时间轴结构，对齐误差稳定控制在 10ms 以内。

### 4. 数据清洗与特征提取的实现

清洗与特征提取以 **异步任务服务**的形式实现，任务由 Celery / Kubernetes Job 调度。

清洗阶段主要包括：

- 基于统计阈值与滑动窗口的异常值检测
- 基于哈希与时间区间的重复数据检测
- 基于姿态连续性的异常跳变检测

特征提取阶段主要包括：

- 使用 OpenPose / MediaPipe 提取 2D 关键点
- 使用 SMPL/SMPL-X 重建 3D 姿态
- 从骨架数据中计算关节角度、速度、加速度
- 从视频中提取环境语义标签（可选）

提取结果以结构化 JSON 或 HDF5 形式存储，并与原始 Asset 形成一对多关系。

### 5. 数据导出的具体实现

数据导出由专门的 **Export Service** 完成。

当用户请求导出时，系统根据标注时间段在 MCAP 文件索引中进行快速定位，只切割相关时间区间的数据块，而不是重新编码整个文件。  
导出结果写入临时对象存储路径，并生成带有效期的签名 URL，确保并发 100 时导出响应时间不超过 3 秒。

## 三、数据标注与审核中心的具体技术实现

### 1. 标注工具的实现细节

标注工具是一个纯 Web 应用，使用 Canvas/WebGL 实现高性能绘制。

视频帧、骨架数据与传感器曲线通过统一时间轴驱动渲染。  
用户的每一次标注操作都会被记录为一条操作日志，并实时写入后端服务。

系统支持自动加载预标注结果，例如由姿态模型或动作分割模型生成的初始标注，标注员只需修正即可。

### 2. 审核与版本控制的实现

每一份标注数据都被视为一个不可变版本。  
修改操作不会覆盖原数据，而是生成新版本，并记录 diff 信息。

所有版本信息存储在 PostgreSQL 中，标注内容本身存储在对象存储中，确保历史可追溯。

### 3. 标准化词典的技术实现

词典服务是一个独立微服务，提供查询、校验与统计接口。  
标注工具在提交标注时，必须通过词典服务进行校验，确保使用的动作与对象符合标准定义。

## 四、可视化与回放系统的具体实现

可视化系统基于 Three.js 构建三维场景，通过 WebSocket 接收后端推送的数据流。

视频播放采用 Video.js，骨架动画与视频帧通过时间戳同步驱动。  
关节角度实时由前端计算或由后端预计算后推送。

## 五、任务管理、权限与数据资产的实现

任务管理、权限控制与数据资产管理集中在一个 **Management Service** 中实现。

权限系统基于 RBAC，实现方式为在数据库中维护角色-权限映射表，并在 API 层通过拦截器进行校验。  
数据资产采用“原始数据 → 预处理 → 标注 → 审核”的状态机模型，所有状态变化均被记录。

## 六、系统底座、监控与安全实现

所有服务运行在 Kubernetes 集群中，通过 HPA 实现自动扩缩容。  
系统监控通过 Prometheus 采集指标，通过 Grafana 展示，通过 Alertmanager 进行告警。

数据传输使用 HTTPS + TLS，数据存储使用 AES-256 加密。  
所有关键操作写入审计日志，满足合规要求。

## 业务架构设计

### 1. 设计目标与原则

本平台面向具身智能训练数据的规模化生产与高质量交付，业务架构设计以**数据资产全生命周期管理**为核心，围绕“**采集 → 处理 → 标注 → 审核 → 展示 → 交付**”这一主线进行规划。

### 2. 业务架构总体分层

平台整体业务架构采用“五层数据业务主线 + 一层横向平台支撑”的结构设计，如下所示：

- 第一层：数据采集层
- 第二层：数据预处理层
- 第三层：数据标注层
- 第四层：数据展示层
- 第四层：数据交付层
- 横向支撑层：平台管理与协同支撑层

前五层构成具身智能数据从生产到交付的完整业务流水线；横向支撑层不直接参与具体数据处理业务，但为各业务层提供统一的管理、协同与安全保障能力。

### 3. 各业务层功能说明

#### 3.1 数据采集层

数据采集与接入层是平台的数据入口，面向真实居家与体育运动等应用场景，负责多模态具身智能数据的采集、接入与上传管理。

该层主要包括以下业务功能：

- **多模态设备接入**：支持相机、IMU、姿态传感器、麦克风等多种采集设备的统一接入与管理；
- **采集模式管理**：支持用户示教采集、人机遥操作采集以及机器人自主采集等多种采集模式；
- **实时与离线采集上传**：支持在线实时数据采集与离线本地缓存，具备断点续传能力；
- **多源数据导入**：支持MCAP、BAG、视频、音频、图片等多种格式数据的批量导入与同步接入。

#### 3.2 数据预处理层

数据预处理与对齐层负责对原始采集数据进行加工与标准化处理，将其转化为适合标注与下游消费的统一数据形态。

该层主要包括以下业务功能：

- **数据清洗**：对采集数据进行去噪、异常值检测与重复数据剔除；
- **格式转换**：将异构原始数据转换为统一的数据格式与数据结构；
- **数据对齐**：基于时间戳对多模态数据进行高精度对齐，保证不同模态之间的时序一致性；
- **特征提取**：提取人体姿态、运动轨迹及环境相关特征。

#### 3.3 数据标注与质量审核层

数据标注与质量审核层是平台的数据价值生成核心，通过对数据进行语义标注与质量控制，将数据转化为高价值的具身智能训练资产。

该层主要包括以下业务功能：

- **智能标注工具**：支持人体姿态、动作序列、目标对象、语音语义等多类型标注；
- **协同标注与任务分发**：支持多标注员协同作业，自动分发与回收标注任务；
- **审核流程与版本管理**：提供“提交—审核—通过/退回”的标准化审核流程，支持标注版本追溯；
- **多模态可视化验证**：通过视频、人体骨架、传感器数据与标注信息的同步回放，辅助质量检查。

该层确保标注数据在准确性、一致性和可追溯性方面满足具身智能模型训练要求。

---

#### 3.4 数据交付与消费层

数据交付与消费层面向模型训练、算法评测与下游应用系统，负责将高质量数据资产以标准化形式对外交付并支撑数据消费。

该层主要包括以下业务功能：

- **标准化数据导出**：支持将数据导出为MCAP、JSON、LeRobot等标准格式；
- **数据切片与版本选择**：支持按时间段、动作区间及标注版本进行数据选择与切割；
- **数据下载与接口服务**：提供安全可控的数据下载与访问接口；
- **下游训练与应用消费**：支撑具身智能模型训练、仿真验证及应用系统的数据消费需求。

该层是数据价值释放的最终环节，实现数据从“资产”到“能力”的转化。

---

### 4. 平台治理与协同支撑层

平台治理与协同支撑层作为横向支撑能力，贯穿于数据全生命周期，为各业务层提供统一的平台级保障。

该层主要包括以下功能：

- **任务与流程管理**：支持项目、任务及子任务的层级化管理与流程跟踪；
- **角色与权限控制（RBAC）**：基于角色的权限管理与访问控制机制；
- **数据资产与版本管理**：对数据在不同阶段形成的资产进行统一管理与追溯；
- **系统监控与安全审计**：对设备状态、系统运行及关键操作进行监控与审计；
- **存储与备份管理**：提供分层存储、数据备份与灾难恢复能力。

该层不直接参与具体的数据处理业务，但为平台实现规模化运行、安全合规与长期稳定提供关键支撑。
