# 平台 Pipeline 与脚本处理输出一致性分析

## 背景

G1 遥操作数据从原始采集到可供 VLA（Vision-Language-Action）模型训练，需要经历两个处理步骤。第一步是**数据合并**（G1_MERGE_CAMERA_ROBOT），将机器人关节数据（HDF5 格式）与 ZED 双目相机图像（SVO2 格式）按时间戳对齐后合并为一个 HDF5 文件。第二步是**格式转换**（G1_CONVERT_TO_LEROBOT），将合并后的 HDF5 转成 LeRobot 标准训练格式（Parquet 文件）。

这两个步骤最初是以独立 Python 脚本（`merge_data.py` 和 `convert_to_lerobot.py`）的形式存在的，后来被封装为平台 Worker 中的 Pipeline 以便自动化调度。本报告对比两种路径在相同输入数据下的输出是否一致，并评估差异对下游训练任务的影响。

测试所用的输入数据为同一份 `episode_0.hdf5`（346 帧机器人关节数据，MD5: `6cd476fc`）和 `episode_0.svo2`（ZED 双目相机录制，MD5: `548fab1e`），两份源文件在两种处理路径中完全一致。

## 对比结果总览

| 维度 | 平台 Pipeline 输出 | 脚本输出 | 是否一致 |
|------|-------------------|----------|----------|
| 帧数 | 346 | 346 | ✅ 一致 |
| 列名与结构 | 11 列，相同 schema | 11 列，相同 schema | ✅ 一致 |
| action 数据 | 逐帧完全一致 | 逐帧完全一致 | ✅ 一致 |
| 状态数据 | 逐帧完全一致 | 逐帧完全一致 | ✅ 一致 |
| 时间戳 | 逐帧完全一致 | 逐帧完全一致 | ✅ 一致 |
| 图像尺寸 | 720×1280×3 | 720×1280×3 | ✅ 一致 |
| 图像内容 | — | — | ⚠️ 同一帧，微小像素差异 |
| LeRobot 元数据 | `_type: "List"` | `_type: "Sequence"` | ⚠️ 不同 |

## 图像差异详解

两种路径产出的图像在像素层面并非完全一致，但这并不意味着它们来自不同的帧或不同的相机。实际上，两边的图像具有几乎相同的统计特征——均值都是 125.3 左右，标准差都是 48.6——说明它们描述的是完全相同的场景。

具体来看，对同一帧画面逐像素对比，绝对差的中位数仅为 **1**（在 0 到 255 的色阶中），83% 的像素差值小于 5，90% 的像素差值小于 10，没有一个像素的差值超过 50。这种量级的差异是完全一致的图像经过不同 JPEG 编解码器后产生的典型误差——肉眼不可见，任何图像处理流程都不会因此产生行为差异。

造成这个差异的根源是，两条路径各自运行在独立的 Python 环境中，环境中的 OpenCV（cv2）和 LeRobot 库版本可能不同。同样的 `cv2.imencode('.jpg', quality=95)` 调用，在 OpenCV 4.8 和 4.10 之间产出的 JPEG 字节流天然存在微小差异——JPEG 是一种有损压缩，其编码器的底层实现细节（如 DCT 系数的舍入策略）在不同版本间会有调整，但解码后的视觉结果是一致的。

## 元数据差异详解

LeRobot 格式在 Parquet 文件的 schema 元数据中记录了每个字段的类型信息。平台输出使用 `_type: "List"` 来描述固定长度数组（如 14 维的 action_eef），而脚本输出使用 `_type: "Sequence"`。这是 LeRobot 库从 v1 到 v2/v3 的格式变更——`List` 是 HuggingFace Datasets 库后期引入的规范类型，语义与 `Sequence` 完全等价，只是名称不同。

## 对 VLA 训练的影响

**没有影响。**

VLA 模型在训练时读取的是解码后的像素数组和浮点数向量，而不是 Parquet 文件中的原始 JPEG 字节或元数据字段名。具体而言：

**Action 标签**是训练的监督信号，两条路径的输出在这一项上逐帧完全一致。只要 action 数据没有差异，模型学到的策略就不会因为处理路径不同而改变。

**图像输入**在进入模型前通常会经过一系列数据增强操作——随机裁剪、色彩抖动、高斯噪声、随机翻转等。这些增强引入的像素扰动通常在 10 到 50 个色阶，远超我们看到的 1 个色阶的中位差。模型根本感知不到这个级别的输入差异。

**元数据字段名**（`List` vs `Sequence`）仅在数据加载阶段被使用。如果训练代码依赖了特定版本的 LeRobot 格式假设，在加载数据时可能因为字段名不匹配而抛出异常。但这是代码兼容性问题，不是数据质量问题，用一个 `if` 判断或者统一 LeRobot 版本即可解决。

## 是否需要追求字节级完全一致

**不需要。**

追求两个处理路径输出完全一致的 Parquet 文件，意味着要让一模一样的输入在两条路径上经过完全相同的 JPEG 编码器、完全相同的 LeRobot 序列化逻辑，产出行级甚至字节级一致的二进制文件。这需要锁定以下所有变量：

1. **OpenCV 版本**完全一致（JPEG 编码器版本相同）
2. **LeRobot 版本**完全一致（序列化逻辑和元数据格式相同）
3. **Python 版本**和 **NumPy 版本**一致（浮点运算结果可能受 MKL/BLAS 后端影响）
4. 以上所有依赖的**底层 C 库版本**也需一致

实现这个目标的唯一办法是将所有依赖锁定为精确版本并统一部署环境。这在实际工程中代价很高——一旦某个依赖发布了安全补丁或 Bug 修复，升级就意味着"打破一致性"。长期来看，维护这种一致性需要冻结整个依赖树，得不偿失。

更重要的是，**这种一致性对于训练没有任何意义**。VLA 模型关心的是语义等价而非二进制等价。只要 action 数据完全一致、图像内容在视觉上一致，模型训练的结果就不会因为处理路径的不同而产生实质偏差。

## 平台处理 Pipeline 的完成度评估

当前平台的 G1_MERGE_CAMERA_ROBOT 和 G1_CONVERT_TO_LEROBOT 两个 Pipeline 可以认为是**已完成并可用于生产**的。

理由如下：

第一，**数据完整性已验证**。Pipeline 产出的帧数、列结构、action 标签、状态向量均与脚本输出完全一致，没有丢失任何数据。

第二，**图像质量与脚本输出等价**。图像差异仅为 JPEG 编码层面的微小扰动，不会对模型训练产生任何可测量的影响。

第三，**Pipeline 逻辑与脚本逻辑一致**。核心算法（时间戳对齐的二分查找、delta EEF 计算、delta height 计算、图像解码）在两个路径中的实现是相同的，没有逻辑偏差。

第四，**平台带来的收益远大于微小差异**。Pipeline 接入平台后，处理任务的创建、调度、产物管理全部自动化，不再需要手动运行脚本和追踪中间文件。Worker 支持断点续传和失败重试，大幅提升了批量处理的可靠性。

## 建议

如果下游训练代码对 `_type` 字段有严格要求，可以在 G1_CONVERT_TO_LEROBOT Pipeline 中固定 LeRobot 版本（在 `requirements.txt` 中锁定版本号），确保所有产物的元数据格式统一。但这不是数据层面的问题，而是环境管理的最佳实践。
