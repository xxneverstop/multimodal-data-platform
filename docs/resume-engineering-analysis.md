# 多模态数据基础设施平台（MMDP）— 工程价值分析

## 项目定位

面向实验室多模态采集场景（双目相机、IMU、RGB-D、动捕等），将"分散存储、命名混乱、人工脚本处理、结果难追溯"的现状，改造为**从采集任务管理 → 标准 Session 导入 → Profile 校验 → 离线 Pipeline 处理 → 资产血缘追溯**的完整数据基础设施。

技术栈：Spring Boot 3 / MyBatis-Plus / MySQL / Vue3 / TypeScript / Python Worker / 阿里云 OSS / Docker Compose

---

## 简历描述（6 条）

---

### 1. 数据全生命周期建模与平台化

> 面向实验室多模态采集数据分散存储、命名不统一、处理依赖人工脚本、结果难追溯等问题，设计并实现覆盖"采集任务管理 → 标准 Session 导入 → 原始文件归档 → Profile 校验 → 离线处理 → 结果资产回放"的数据基础设施平台，将实验数据流转流程平台化、结构化和可追踪化。

**背后的设计：** 核心思路是用一套统一的数据模型把原本散落在各个硬盘、各种命名习惯里的实验数据"管起来"。模型从顶层到底层分为三层：最上层是采集任务——它对应一次实验行动（比如"张三-走路-6月1日"）；中间层是采集会话——每次按下录制按钮产生一个会话，一个任务可以包含多次录制；底层是数据资产——它是对所有数据的统一抽象，不管这个数据是手动上传的文件、采集设备导入的原始数据、还是后续处理产生的结果。资产的来源有三种，但对下游来说都是同一套接口，不关心它从哪来。

这种分层的好处是，后续的血缘追踪、Pipeline 匹配、回放组装都基于这套模型，不需要每次加新功能就重新理解数据结构。同时设计了三级数据清理能力——可以只删一个处理任务的产物，也可以删整个采集会话，或者删整个任务的全部数据——数据库记录和云存储上的文件同步删除，不留孤儿数据。

### 2. Profile 驱动的多模态数据接入机制

> 设计 Profile 驱动的多模态数据接入机制，将不同设备组合下的必需文件、sourceKey、资产类型、校验规则与处理入口从主流程中拆分，使 ZED 双目相机 + IMU、双路视频 + IMU、RGB-D 等异构采集组合可以通过统一 Profile 规则描述，降低新增设备组合时对核心导入逻辑的侵入。

**背后的设计：** 这里要解决的核心问题是"实验室设备组合经常变，但导入流程不应该跟着变"。一个 Profile（采集配置模板）用声明式的方式描述了一类采集场景的四个规则：第一，打包规则——上传的 ZIP 包里必须有哪些文件才算完整；第二，解析规则——拿到一个文件名，怎么判断它是左摄像头的数据还是 IMU 的数据；第三，归档规则——这些文件在云存储上按什么目录结构存放；第四，回放规则——前端播放时，哪些数据是视频、哪些该画曲线图。

这四个规则各自是一个接口，具体的设备类型去实现它。导入流程只依赖接口，不依赖具体设备。新增一种设备组合，就是新增一个规则实现类放在项目里，然后在数据库里创建一条 Profile 配置指向它——导入流程的代码一行都不用改。目前项目支持了通用设备、伪双目立体等好几套规则，每套规则的实现都在几十到一百多行，改动范围完全隔离。

为什么不用规则引擎或者脚本语言来做？因为当前的规则逻辑都很简单——判断文件名后缀、检查文件列表里有没有缺失某个 source——每个规则只有几十行代码，用一种编程语言写最直接。引入 Drools 或自己设计 DSL 的学习成本和运维成本远超收益。等到规则复杂到需要嵌套条件、优先级排序的时候，再抽象也来得及。

### 3. 标准 Session 导入与幂等处理

> 设计标准 Session 目录导入与幂等处理方案，通过 manifest.json 描述一次采集的设备来源、文件路径、时间戳和 Profile 信息，并在 finalize 阶段完成校验、归类、Session 创建、资产登记和处理入口衔接；基于 localSessionId / requestId 支持网络重试场景下的安全重复提交。

**背后的设计：** 采集端在野外环境工作，网络不稳定是常态。最坏的情况是：数据已经上传成功了，但给采集端的 HTTP 响应在网络中丢失了，采集端以为失败，又上传了一次。如果平台不做幂等保护，就会创建两条重复的会话记录和两份重复的文件。

这里的方案利用了采集端生成的本地会话 ID——它是采集端在录制开始时就生成的唯一标识，不受网络影响。平台在收到导入请求后，第一步不是开始处理，而是先拿这个 ID 去数据库里查："我是不是已经导入过这个会话了？"如果找到了，直接把已有的结果返回，后面的校验、解压、归档全部跳过。这个查询依赖数据库的唯一索引来保证——如果两个请求碰巧同时到达，数据库层面会拒绝第二个写入，不会出现两条重复记录。

对于分步上传的场景（采集端先把大文件直接传到云存储，再通知平台去归档），幂等检查用了两个维度：一个是上面说的本地会话 ID，另一个是采集端每次请求生成的追踪 ID。两个维度任一个命中就返回已有结果。

整个导入过程还做了分步持久化——校验完成、文件分类完成、资产创建完成，每一步的状态都写入数据库。如果某一步失败了，运维可以直接看到"卡在归档阶段，原因是某个文件上传 OSS 超时"，不需要从头排查整个流程。

### 4. Java 后端与 Python Worker 解耦的离线处理链路

> 搭建 Java 后端与 Python Worker 解耦的离线处理链路，后端负责任务状态、资产元数据和处理生命周期管理，Worker 从 OSS 拉取源文件并执行格式转换、视频重建等计算任务；已打通"抽帧图像 + IMU CSV → 可播放 MP4"的端到端处理闭环，为后续 IMU 对齐、自动质检和算法 Pipeline 扩展预留接口。

**背后的设计：** 处理任务涉及大量 Python 生态的计算——视频编码依赖 ffmpeg 命令行工具，人体模型推理需要 PyTorch，传感器时间对齐用 numpy/scipy。这些计算不可能放在 Java 后端里跑。所以设计了一个"后端管调度、Worker 管执行"的分离架构。

调度方式选择了轮询而不是消息队列。原因是这个场景的处理任务都是分钟级的——一段视频编码通常要跑几十秒到几分钟，任务的产生频率远低于 Web 请求。在这样的 TPS 下，消息队列的毫秒级推送能力完全用不上，但它带来的运维负担是真实的：需要维护 Broker、处理消息堆积、配置死信队列。而轮询协议只依赖 HTTP，Worker 挂了重启就是，多部署几个实例也能水平扩展，整个系统的运维组件少了一个。

Worker 端的 Pipeline 注册用了自动发现机制：启动时扫描指定目录下所有 Python 文件，找到继承了 Pipeline 基类的子类，自动注册。加一个新的处理脚本只需要写一个 Python 类——定义它需要什么类型的输入、产出什么类型的输出、核心的执行逻辑怎么写——然后重启 Worker 即可。不需要改任何配置文件或注册表。

目前已经打通了 7 个 Pipeline，覆盖了从图像序列合成视频、双目对齐、IMU 数据对齐、3D 动作数据转换、物理指标计算等几个典型场景。每个 Pipeline 声明了它需要什么输入和产出什么输出，后端可以根据一个会话里已有的资产类型，自动判断哪些 Pipeline 可以执行，不需要用户自己猜。

### 5. OSS 大文件直传与存储抽象

> 设计 OSS 大文件直传与存储抽象方案，采用 initiate / complete 两阶段上传语义，由后端签发上传凭证并在上传完成后登记文件元数据，避免大文件经后端中转造成带宽瓶颈；同时保留 StorageService 抽象，便于后续适配不同对象存储后端。

**背后的设计：** 多模态数据的特点就是大——一段 30 秒的双目 4K 视频可以到上百 MB，高频 IMU 采样的 CSV 也能到几十 MB。如果走传统的"浏览器上传到后端，后端再转发到云存储"的路径，带宽要消耗两倍——进来一份，出去一份。10 个采集端同时上传，后端带宽很快打满。

解决方案是让浏览器绕过后端，直接把文件传到云存储。用的是云厂商提供的 STS 临时凭证机制：后端签发一个有时效的凭证（通常 15 分钟过期），限定只能上传到指定的路径。浏览器拿到凭证后直接和云存储交互。整个过程中后端只处理几十字节的 JSON（签发凭证和完成确认），完全不接触文件字节流。

上传流程设计成两步：第一步 initiate——后端生成凭证和上传路径，返回给前端；第二步 complete——前端上传完成后通知后端去确认文件确实在云存储上存在，然后走和同步导入一样的校验、分类、资产创建流程。这种两阶段的语义也方便后续扩展——比如在 initiate 阶段可以加文件大小配额检查，在 complete 阶段可以触发后续的自动处理。

存储层做了接口抽象，虽然目前只用阿里云 OSS，但切换底层存储只需要新增一个实现类，不影响业务代码。这是在"不用过度设计"和"预留扩展点"之间找了一个平衡——接口先定义好，但只有一个实现的时候不做多 Provider 的动态路由。

### 6. 核心业务模型设计与数据底座

> 负责平台核心业务模型设计，将采集流程抽象为 Task、Session、DataAsset、ProcessingJob、QC Report 等实体，建立从采集计划、数据入库、资产管理、处理任务到质检结果的完整业务闭环，为后续数据血缘、自动标注和算法处理扩展提供统一数据底座。

**背后的设计：** 业务模型有三个关键设计决策。

第一个是**统一资产抽象**。平台里的数据来源各异——手动上传的文件、设备导入的原始数据、外部系统的数据引用——但对下游来说，它们都是"数据资产"，有统一的类型标签、统一的来源追踪、统一的展示名称。后续的 Pipeline 匹配、血缘图构建、回放数据组装都基于这个抽象层，不关心资产的具体来源。

第二个是**血缘的三元组设计**。传统的做法是给每个资产加一个"父资产"字段，但实际的处理场景里，一个处理任务可能消费多个输入（左摄像头图像、右摄像头图像、IMU 数据），产出多个输出（对齐视频、对齐报告）。用三元组——源资产、目标资产、处理任务——来表达这段关系，把处理任务作为中间节点，天然适配多对多的血缘关系。沿着这个三元组表向外遍历，就能从任意一个资产还原出完整的数据加工链路。

第三个是**处理任务的状态机**。任务从创建到完成经历了"待处理→已认领→成功/失败"的状态流转。Worker 只认领"待处理"状态的任务，处理后上报成功或失败。这个简单的状态机是整个异步处理调度协议的基础。任务之间还可以声明依赖关系——比如"视频合成"必须等"时间对齐"完成之后才能跑——依赖信息以 JSON 数组的形式存在任务记录里，构建血缘图时把这些依赖也渲染成图上的边。

## 面试问答

以下是面试中简历描述可能引发的追问。回答用口语化语言，重点讲设计思路和取舍逻辑，不罗列代码细节。

---

### Q1："上传幂等具体是怎么实现的？如果两个请求同时到达会怎样？"

**回答：** 幂等保护的核心思路是"先查后做"——收到导入请求的第一步不是开始处理，而是先去数据库里查"这个会话我是不是已经导入过了"。用什么来判断是不是同一个会话呢？用的是采集端在录制一开始就生成的一个本地唯一 ID，它不是平台分配的，是采集端离线生成的。这个 ID 和网络状况无关，无论重试多少次，同一个会话的 ID 不变。

这个 ID 在数据库里有唯一索引约束。这意味着即使有两个请求在极短的时间窗口内同时到达、同时查到"不存在"、同时尝试插入，数据库层面也会拒绝第二个插入，不会产生两条重复记录。

对于分两步上传的场景（客户端先传文件到云存储，再通知平台），幂等检查用了两个维度：除了上面说的本地会话 ID 之外，客户端每次请求还会生成一个追踪 ID。两步上传的第二步如果因为网络问题重试了，追踪 ID 可以拦住重复的"完成确认"请求。

坦率说，当前实现没有用数据库行锁或分布式锁来做并发控制。这是因为实际上同一个采集端的同一个会话几乎不会在毫秒级窗口内并发提交——野外网络的重试间隔远大于这个窗口。如果未来 Worker 数量增加或者出现并发场景，用行锁或者 Redis 分布式锁加固一下就行，改动量很小。

**追问："那你为什么选择数据库唯一索引做幂等，而不是在应用层用 Redis？"**

因为唯一索引是最可靠的"最后一道防线"。不管应用层逻辑有没有 bug，不管有没有并发竞争，数据库这个约束是永远不会被绕过的。Redis 可以加速"是否存在"的判断，但它和数据库之间没有事务保证——Redis 说"不存在"，但就在这几毫秒里另一个请求已经把数据写进数据库了。所以我的思路是：数据库唯一索引作为兜底，Redis 缓存作为可选的性能优化。当前场景下查询量很低，连缓存都不需要，直接查库就行。

---

### Q2："Profile 规则引擎和简单的 if-else 有什么区别？为什么不直接写死在代码里？"

**回答：** 区别在于"变化的东西由谁来改、改的时候会不会动到不该动的地方"。

如果只有两三种设备类型并且永远不会增加，if-else 没问题。但实验室环境的特点是设备组合持续变——这周用 ZED 双目相机加 IMU，下周可能加一套 Xsens 动捕服，再下周换 Intel RealSense。如果把每种设备的处理逻辑写死在导入流程里，每加一种设备就要去改整个平台最核心、最复杂的那段代码——这段代码有近两百行，包含了解压校验、文件分类、资产创建等多个步骤，改动风险很高。

现在这套方案把"会变的部分"隔离成了四个独立的小接口：怎么校验包结构、怎么根据文件名判断它属于哪个数据源、怎么决定存储路径、怎么组装回放数据。导入主流程只依赖这四个接口，不依赖任何具体设备。加新设备就是写一个新的实现类放在项目里，Spring 框架会自动发现它并注册到规则表里。然后在数据库里创建一条配置记录，把新的设备关联到新的规则实现上。整个过程中，导入主流程的代码一行都不需要改。

**追问："这其实就是策略模式对吧？有什么特别的地方吗？"**

对，本质上就是策略模式。特别的地方在于它把"哪个策略用于哪个场景"这个映射关系从代码里移到了数据库里。传统的策略模式通常是代码里用一个 switch-case 或者 Map 来根据 key 选择策略——key 变了或者策略变了还是要改代码。这里的映射是通过数据库的配置记录来维护的，新增设备和新增规则的绑定关系通过一条 SQL insert 完成，不需要重新编译部署。这个变化看似小，但意味着"新增设备类型"这件事可以从开发任务变成配置任务——未来可以做一个管理界面让运维人员自己配，不需要开发介入。

---

### Q3："Worker 用轮询而不是消息队列，任务多了会不会有问题？"

**回答：** 这是有意为之的选择，关键在于这个场景的任务特征：处理任务是分钟级的离线计算——视频编码可能要跑 30 秒，IMU 对齐可能要跑几分钟。任务产生的频率极低，通常每分钟不到一个。

消息队列解决的核心问题是削峰填谷和毫秒级延迟——比如秒杀场景下瞬间几十万请求，用 MQ 缓冲一下避免后端被打垮。但在分钟级离线处理的场景里，这两个能力完全用不上。但消息队列带来的运维成本是真实的：你需要维护一个额外的中间件，要监控 Broker 是否健康，要处理消息积压的告警，要配置死信队列防止消息丢失——整个系统的运维复杂度至少翻倍。

轮询方案的好处就是简单。Worker 和后端之间只依赖 HTTP 协议，Worker 每隔几秒问一次"有活吗"，有就干，没有就等。Worker 进程挂了，重启就行，它自己会继续轮询。想增加处理能力？多部署几个 Worker 实例，它们会自己去竞争认领任务——数据库查询的时候只取最早的一条，谁先拿到谁处理。

**追问："多 Worker 同时认领会冲突吗？"**

当前实现用的是先查询再更新的方式——查到一条待处理的任务，然后把它的状态改成"已认领"。严格来说，如果两个 Worker 在同一个毫秒级窗口内查到了同一条记录，确实可能重复认领。但实际运行中，我们目前是单 Worker 部署，所以这不是瓶颈。如果扩展成多 Worker，改一行代码加上数据库行锁就能解决——查询的时候用 select for update 锁定这条记录，事务提交之前其他 Worker 读不到它。这个改动非常简单，不需要调整架构。

**追问："为什么不用 gRPC 或者其他 RPC 协议？"**

因为 HTTP 够用了。这个场景的通信量极低——Worker 发一个 POST 请求问有没有任务，后端返回一个 JSON。RPC 的优势在于流式传输和强类型接口，但这里的交互模式就是一个简单的请求-响应，用 HTTP 没有任何性能瓶颈。而且 HTTP 的好处是调试方便——用 curl 就能模拟 Worker 的行为，出问题的时候排查更快。

### Q4："Pipeline 的输入/输出类型声明有什么用？给面试官讲一下这个设计。"

**回答：** 它解决的是一个很实际的问题——"用户面对一堆数据，他能做什么处理？"

每个处理脚本都声明了自己需要什么类型的输入数据，以及会产出什么类型的输出数据。比如"图像序列合成视频"这个脚本声明它需要"原始图像序列"作为输入，产出"MP4 视频"作为输出。

当用户打开一个采集会话的详情页，想对数据做些处理的时候，后端做的事情不是把所有 7 个处理选项全列出来让用户自己猜，而是先去查这个会话里已经有哪些类型的数据，然后逐个匹配：哪些处理脚本需要的输入数据我都已经有了？只把匹配上的那几个展示给用户。

这个设计还有一个隐性的好处——它让处理脚本之间可以链式串联。一个脚本的输出类型（比如"对齐后的 MP4 视频"）可以成为另一个脚本的输入声明（比如"制作播放包"需要的就是 MP4 视频）。系统不需要预先定义好固定的处理流程，而是通过类型匹配自动发现可执行的链路。

**追问："那怎么防止用户乱点，比如先跑合成视频再跑对齐，顺序反了？"**

处理任务之间支持显式依赖声明——创建任务的时候可以指定"这个任务得等哪几个任务完成之后才能开始"。这层依赖关系存在任务记录里，Worker 认领任务时并不强制检查依赖（当前是用户自己在界面上判断），但血缘图会把依赖关系渲染成图上的边，用户可以直观地看到处理链路。

### Q5："你说的资产血缘，和直接给每个资产加一个 parent 字段有什么区别？选哪个更好？"

**回答：** 区别在于"一对一"和"多对多"。

parent 字段的方案只能表达"这个资产是由哪个资产产生的"——这是一对一或一对多的树形结构。但在实际的处理场景里，一个处理任务通常会消费多个输入（比如时间对齐这个任务同时需要左眼图像序列、右眼图像序列和 IMU 数据），产出多个输出（对齐后的视频和对齐报告）。parent 字段表达不了"这三个输入一起产生了这两个输出"。

我用的方案是用一张独立的血缘表，三元组——哪个源资产、经过哪个处理任务、变成了哪个目标资产。把处理任务作为中间节点，天然表达多对多的关系。而且这种设计让查询非常灵活：从任意一个资产出发，沿着血缘表往前查能追溯到它是由谁产生的、用了什么参数；往后查能看到它被哪些后续处理消费了。整个数据加工链路是一条完整的图，不是一棵树。

**追问："这个三元组的设计是参考了什么？"**

参考了数据仓库里"数据血缘"的概念，做了大幅简化。数据仓库里的血缘通常要追踪到字段级别（这一列来自哪个表的哪一列、经过了什么转换），但对于多模态数据来说，资产级别就够用了——我只需要知道"这段视频是由这个会话的原始图像序列合成的"，不需要知道视频里的每一帧对应哪个原始文件。设计关键是把粒度控制在能回答业务问题的程度，而不是为了"完整"而过度设计。

### Q6："数据量大了以后，上传 200MB 的文件，后端内存会不会炸？"

**回答：** 大文件上传走的是直传模式，后端不接触文件字节流。

具体的做法是：前端在上传之前先向后端申请一个上传凭证，后端调用云存储的 STS 接口，签发一个有时效的临时凭证——里面包含临时的访问密钥和安全令牌，限定只能上传到我们指定的路径。前端拿到这个凭证后，用云存储提供的 JS SDK 直接把文件传到云上。整个过程中后端只处理了几十字节的 JSON，不管文件是 1MB 还是 200MB，后端的内存消耗是一样的。

上传完成后，前端通知后端"我传完了"，后端用 HTTP HEAD 请求去云存储确认一下文件确实存在，然后走正常的元数据登记流程。

**追问："为什么要分两步？为什么不直接让前端传完就完事？"**

因为后端需要"确认"这件事——如果前端传了一半失败了、或者传错了路径、或者根本没传，后端需要知道。HTTP HEAD 请求是一个极轻量的校验，只返回文件的元数据不返回内容，能在几乎零成本的情况下确认文件确实到位了。有了这个确认步骤，后续的处理流水线才能放心地去拉这个文件。

**追问："如果是同步导入，采集端直接上传到后端一个几百兆的 ZIP 包呢？"**

坦率说，当前同步导入的实现是把 ZIP 包全部读进内存再逐文件解压处理的。这是因为需要随机访问 ZIP 里的各个文件来分类和校验。目前实验室场景下的 ZIP 包通常在 50 到 100MB，一次性加载在内存里问题不大。如果后续遇到超过 200MB 的包，改进方向是流式解压——读到一个文件就处理一个、上传一个到云存储，然后释放内存。这属于已知的优化点，当前场景还没有触发。

### Q7："Profile 里有个 playback_rule，为什么要单独抽象出来？它做了什么？"

**回答：** 它回答的问题是"前端回放数据的时候，哪些数据能播、以什么形式播、播哪个文件"。

不同类型的设备，回放方式完全不同。举个例子：一个 ZED 双目相机的采集会话，回放时左眼一个视频窗口、右眼一个视频窗口、IMU 数据画成曲线图显示在下方——三个播放源，两种播放形式。而一个伪双目的场景可能只有一个视频源但要左右分屏显示。如果用硬编码的方式，前端需要知道每种设备的播放逻辑，每次加新设备前后端都要改。

这个规则做的事情就是把"怎么播"的逻辑封装在后端：输入是一个会话 ID，输出是一个"播放清单"——告诉前端，这儿有一个视频，地址是这个；这儿有一组 IMU 数据，画曲线图。前端拿到这个清单后只需要遍历渲染，不需要理解设备差异。这样新增一种设备的回放方式，只需要后端加一个规则实现，前端不需要改代码。

**追问："这不就是一个接口？有什么值得说的？"**

对，就是一个接口，但关键在于它是从原来几百行的回放组装方法里拆出来的。原来的做法是在一个方法里用 if-else 判断设备类型、拼接不同的数据源，每加一种设备这个方法就膨胀一次。拆成接口加策略实现之后，每种设备的回放逻辑在自己的类里，互不影响，主方法只负责调用规则获得结果，然后返回给前端。这是典型的"把变化隔离出去"的思路。

### Q8："这个项目你一个人做的？你在里面做了哪些独立的技术决策？"

**回答：** 这个项目代码层面是我独立完成的。在做的过程中有几个关键决策是我权衡之后自己定的：

第一个是**数据模型怎么设计**。12 张表的关系、特别是把数据资产作为统一抽象层和用三元组做血缘追踪，这些是参考了数据仓库中数据血缘的理念，然后针对多模态数据的实际特点做了简化。核心原则是"模型要能回答业务问题，但不要为了完整而过度设计"。

第二个是**为什么用策略模式而不是规则引擎**。当前设备类型的规则逻辑都很简单——判断扩展名、检查文件列表是否完整、根据固定模式匹配文件名——每个规则几十行代码。这种复杂度下用 Drools 这种重型规则引擎，带来的学习成本和运维负担远超收益。但我没有把规则写死，而是用了接口加自动注册的方式，保留了未来规则的扩展性——接口已经在，真到了需要表达式引擎的时候，加一个"表达式规则实现类"就行。

第三个是**Worker 调度为什么用轮询而不是消息队列**。这是在评估了实际场景后做的取舍——处理任务都是分钟级的离线计算，TPS 极低，消息队列的能力用不上，但运维成本是躲不掉的。轮询方案简单可靠，系统少一个中间件就少一个出故障的点。

第四个是**存储层的接口抽象**。虽然目前只用阿里云 OSS，但我在接口层面就做了 Provider 抽象，切换底层存储不需要改业务代码。这是"不过度设计但预留扩展点"的典型做法。

需求层面的方向——比如要支持哪些设备、Pipeline 的业务流程应该怎么走——是和实验室团队讨论确定的，我会根据他们的反馈调整设计。

## 架构决策记录

| 决策点 | 选择了 | 没选 | 为什么 |
|---|---|---|---|
| Worker 调度 | HTTP 轮询 | 消息队列 | 处理任务是分钟级离线计算，TPS 极低，MQ 运维成本远超收益 |
| 设备规则 | 策略模式 + 接口自动注入 | 硬编码 if-else 或规则引擎 | 规则简单但变化频繁，轻量策略模式刚好；规则引擎太重 |
| 存储后端 | OSS 单实现 + 接口预留 | 多 Provider 并行 | 接口已预留扩展点，暂无多后端需求，不过度设计 |
| 数据库 ORM | MyBatis-Plus | JPA/Hibernate | 血缘图构建等场景需多表批量查询，SQL 可控比自动映射更重要 |
| 认证方案 | Session + 拦截器 | JWT | 单体应用，Session 更简单，不需要刷新 token 和分布式会话 |
| Pipeline 注册 | Python 反射自动扫描 | 手动注册清单 | 消除"新增脚本忘记注册"的人为错误 |
| 前端 UI | 自建设计系统 | Element Plus / Ant Design | 项目早期 UI 定制需求多，组件库反而限制灵活性 |
