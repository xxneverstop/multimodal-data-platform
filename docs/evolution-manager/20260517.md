# 2026-05-18 版本总结

## 一、版本定位

当前仓库已经完成多模态数据平台第一阶段 MVP 的主干闭环，重点围绕 IMU 数据采集场景，实现了：

`采集任务创建 -> 原始文件上传 -> OSS 对象存储 -> 文件元数据入库 -> 同步执行基础 QC -> QC 报告入库 -> 前端查询与展示`

从工程完成度来看，这一版已经不是纯原型文档，而是具备前后端代码、数据库表结构、对象存储接入、基础质检逻辑和管理界面的可运行雏形。

## 二、当前已实现功能

### 1. 采集任务管理

- 支持创建采集任务。
- 支持分页查询任务列表。
- 支持查看单个任务详情。
- 任务元数据已覆盖任务名、被试编号、动作名、设备类型、模态、采集日期、备注等字段。
- 任务状态已建立基础流转：`CREATED`、`UPLOADED`、`QC_PASSED`、`QC_WARNING`、`QC_FAILED`。

对应后端接口：

- `POST /api/tasks`
- `GET /api/tasks`
- `GET /api/tasks/{taskId}`

### 2. 文件上传与对象存储

- 支持按任务上传原始数据文件。
- 上传文件后会写入 `data_file` 表，保存文件名、扩展名、大小、内容类型、桶名、对象键、存储路径、上传状态等信息。
- 文件内容实际写入 OSS。
- OSS bucket 支持自动检查与自动创建。

对应后端接口：

- `POST /api/tasks/{taskId}/files`
- `GET /api/tasks/{taskId}/files`
- `GET /api/files/{fileId}`

### 3. 基础 QC 质检

- 文件上传完成后，后端会同步触发 QC 检查，无需额外调度。
- QC 结果写入 `qc_report` 表。
- QC 报告同时保留摘要字段和结构化 `report_json`。
- 当前 QC 总状态分为：`PASSED`、`WARNING`、`FAILED`。

当前已实现的质检规则包括：

- 通用规则
  - 文件非空检查
  - 扩展名是否在 MVP 支持范围内
  - 文件大小是否过小或过大
- `txt` 时序样例检查
  - 是否可读取样例行
  - 第一列是否类似时间戳
  - 抽样行列数是否一致
  - 数值列是否可解析为数值
- `csv` 检查
  - 是否至少包含表头和一行数据
  - 是否存在 `timestamp` 表头
  - 是否存在 `acc` / `gyro` 相关字段
- `json` 检查
  - JSON 语法是否可解析
  - 当前仅做语法校验，尚未做业务结构校验
- `bvh` / `fbx`
  - 当前仅判定扩展名支持，并给出“内容检查有限”的 warning

对应后端接口：

- `GET /api/tasks/{taskId}/qc-report`

### 4. 统一响应与异常处理

- 后端 API 已统一封装为 `ApiResponse<T>`。
- 已实现全局异常处理，覆盖业务异常、参数校验异常、JSON 解析异常、文件缺失、上传超限、数据库异常等常见场景。
- 这意味着前端已经可以按统一格式接收成功或失败响应。

### 5. 数据层与基础工程配置

- 已建立三张核心表：
  - `acquisition_task`
  - `data_file`
  - `qc_report`
- 后端技术栈已落地：
  - Java 21
  - Spring Boot 3.3
  - MyBatis-Plus
  - MySQL
  - OSS SDK
- 前端技术栈已落地：
  - Vue 3
  - Vue Router
  - TypeScript
  - Vite
  - Axios
  - Tailwind CSS 4

侧边栏和路由中已经预留以下模块入口，但当前仍是占位页：

这说明当前前端的重点是“任务-上传-QC 报告”主链路，平台化的横向模块尚未进入实装阶段。

### 4. 前端当前问题

- 前端 `http.ts` 默认 API 地址为 `http://localhost:8080`。
- 后端 `application.yml` 当前端口配置为 `19021`。
- 如果没有额外 `.env` 覆盖，前后端默认配置存在端口不一致问题，前端直连后端时会请求失败。

因此，前端界面虽然已经接好了 API 调用代码，但本地联调依赖额外修正 API base URL。

## 四、后端已实现链路

### 1. 任务链路

后端已经完整实现任务的创建、查询、详情获取，并以 `AcquisitionTaskServiceImpl` 负责状态更新。

当前任务状态流转逻辑为：

`CREATED -> UPLOADED -> QC_PASSED / QC_WARNING / QC_FAILED`

### 2. 文件上传链路

文件上传链路已经闭环，执行顺序如下：

1. 校验任务存在。
2. 校验上传文件非空。
3. 解析原始文件名、扩展名、内容类型。
4. 生成 OSS `objectKey`。
5. 上传到 OSS。
6. 更新任务状态为 `UPLOADED`。
7. 写入 `data_file` 元数据。
8. 调用 `QcInspectionService` 做同步质检。
9. 写入 `qc_report`。
10. 根据 QC 结果把任务状态更新为 `QC_PASSED` / `QC_WARNING` / `QC_FAILED`。
11. 返回包含文件信息、QC 状态、摘要和报告 JSON 的上传响应。

这是当前版本最完整的一条业务主链路。

### 3. QC 报告查询链路

QC 报告查询链路也已经贯通：

1. 前端按任务 ID 请求报告列表。
2. 后端校验任务存在。
3. 从 `qc_report` 表读取该任务下的报告列表。
4. 将数据库中的 `report_json` 反序列化为 JSON 对象。
5. 返回给前端做结构化展示。

### 4. 数据存储分层

当前存储职责划分比较清晰：

- MySQL 负责结构化元数据
  - 任务信息
  - 文件元数据
  - QC 报告摘要与 JSON
- OSS 负责原始文件对象存储

这套分层已经满足后续扩展到更多模态文件的基本要求。
