# 2026-05-22 MMDP 前后端当前状态记录

## 一、文档目的

本文用于记录截至 `2026-05-22` 当前仓库内前端与后端的真实状态，重点说明：

- 当前已经完成并可运行的能力
- 当前仍属于前端聚合占位、Mock 或 MVP 级实现的部分
- 前端页面结构与 UI 当前落地情况
- 后端 API、数据模型与处理链路当前落地情况
- 下一阶段继续演进时需要关注的边界

---

## 二、项目总体状态

当前仓库仍处于 `MVP 持续扩展阶段`，但已经不再只是原型或静态页面。

现状可以概括为：

- 后端已经具备任务、文件、资产、基础 QC、可用处理流程判断、处理作业、数据链路等一批真实接口与真实落库能力
- 前端已经从原始任务页，重构为“主页 / 功能 / 管理”的控制台式结构
- 前端已经围绕“数据资产”组织主要页面，而不是围绕“文件列表”
- 首页、数据、上传、采集、处理、标注、质检、导出、管理等页面都已经落地为可运行页面
- 处理、标注、质检、导出、首页趋势图等部分仍有明显前端聚合或 mock 成分

一句话总结：

> 当前 MMDP 已经具备“真实业务骨架 + 真实部分接口 + 前端控制台工作台”的形态，但算法处理、深度质检、真实 Session 主表、真实首页统计仍未完全落地。

### 6. 前端当前的数据组织方式

当前前端已经按新的产品语义组织页面：

- 主实体视角是 `数据资产`
- 页面强调 `taskId` 与 `sessionId` 两个上下文维度
- 数据页主模型为资产，不再是文件
- 处理页主视角为规则 / 模板，而不是单次执行
- 导出页主视角为“成品资产视图”

但底层兼容层仍存在：

- Session 仍由前端基于任务推导
- 首页统计由前端从 task / asset / job / report 聚合得出
- 部分状态与趋势并非后端真实统计接口返回

### 7. 前端当前主要风险

前端当前最大的几个风险点是：

- 路由与页面已经超前于后端实体模型，存在“页面语义已升级、后端仍是兼容推导”的情况
- `sessions` 路由仍存在，但用户侧菜单已隐藏，后续是否继续保留需要产品明确
- 首页趋势图和分布图虽然可运行，但当前不适合作为真实经营数据看板
- 多处历史页面和旧文案仍在仓库中保留，存在后续继续清理和统一的空间

## 四、后端当前状态

当前后端已经存在的主要接口包括：

- `POST /api/tasks`

- `GET /api/tasks/{taskId}/processing-jobs`
- `GET /api/tasks/{taskId}/lineage`
- `GET /api/processing-jobs/{jobId}`

这些接口已经覆盖了当前 MVP 中的主要真实链路。

### 2. 后端当前真实业务骨架

后端当前已经形成以下核心业务骨架：

- 业务任务 `acquisition_task`
- 文件 `data_file`
- 数据资产 `data_asset`
- 质检报告 `qc_report`
- 处理作业 `processing_job`
- 处理流程可用性判断 `available-pipelines`
- 基础链路视图 `lineage`

换句话说，当前后端已经不只是“传文件 + 看报告”，而是具备了一层真实的数据资产和处理作业骨架。

### 3. 后端当前真实可运行链路

当前可以认为是真实闭环的后端链路包括：

#### 3.1 任务创建链路

- 可创建真实任务
- 数据真实落表
- 任务字段已支持扩展后的采集业务字段

#### 3.2 文件上传链路

- 文件真实上传到 MinIO
- 文件元信息真实写入 `data_file`
- 上传后会自动创建对应资产 `data_asset`
- 会同步触发基础 QC
- 会生成真实 `qc_report`

#### 3.3 外部资产登记链路

- 可以只登记外部路径资产
- 不经过 MinIO
- 不生成 `data_file`
- 会真实写入 `data_asset`

#### 3.4 任务资产列表链路

- 能查询某任务下的全部资产
- 包含上传资产与外部登记资产
- 是真实数据库结果，不是前端 mock

#### 3.5 可用处理流程判断链路

- 能根据任务已有资产类型判断处理流程是否具备执行条件
- 当前属于真实的“存在性判断”
- 不是纯前端写死

#### 3.6 处理作业链路

- 处理作业可真实创建
- 处理作业状态可真实查询
- 作业详情可真实查询
- 手工输出资产模式也已具备接口

### 4. 后端当前仍属于 MVP / 占位的部分

虽然处理作业链路已经有接口和状态流转，但当前仍有明显占位实现：

#### 4.1 Session 主实体未真正落地

- 当前前端围绕 `sessionId` 组织了很多页面
- 但后端目前还没有真正独立的 `session` 主表与对应接口
- 现阶段更多是由任务维度兼容推导出的会话概念

#### 4.2 处理算法执行仍非真实算法结果

- `processing_job` 结构、接口和状态流转是真实的
- 但实际算法执行仍偏 mock / 占位
- 当前更像“真实作业壳层 + 占位处理结果”

#### 4.3 QC 仍以基础文件检查为主

- QC 执行是真实的
- 报告落表是真实的
- 但规则仍主要是基础格式/结构检查
- 还不是深层业务质检或多模态一致性质检

#### 4.4 首页与管理类聚合接口缺失

- 后端目前没有专门的首页看板接口
- 没有专门的标注、导出、质检规则、管理配置聚合接口
- 当前这些页面大多依赖前端聚合与 mock 数据

#### 4.5 资产链路仍未完全进入“多文件 / 多派生”形态

- 当前 `data_asset` 已存在
- 但距离“一个资产关联多个文件”“派生资产完整闭环”“可交付资产统一视图”还有差距

### 5. 后端当前技术栈状态

当前后端栈从代码和 README 可确认包括：

- Java 21
- Spring Boot 3
- MyBatis-Plus
- MySQL
- MinIO

当前 README 仍强调的是最初 MVP 闭环：

`create task -> upload file -> persist data_file -> QC -> qc_report`

说明：

- 仓库代码实际能力已经比 README 描述更往前一步
- 但 README 与 manager 文档还存在一定滞后

### 6. 后端当前主要风险

当前后端的主要风险点是：

- 业务模型已向 `task -> session -> asset -> job -> report` 演进，但数据库和接口还未完全跟上
- 前端已经在产品层面引入 `sessionId` 维度，后端没有真实 Session 表会持续造成前后语义错位
- 首页统计、标注、质检规则、导出视图缺少专用后端聚合接口
- 处理作业存在“状态真实、结果占位”的结构性不一致

## 五、前后端配合关系的当前判断

### 1. 当前配合是“前端先行，后端骨架跟进”

从当前代码状态看，系统已经明显进入：

- 前端产品结构先重构
- 后端继续保持 MVP 真实链路
- 前端通过兼容聚合把产品语义先铺开

这种方式的好处是：

- 可以先把平台结构、页面骨架、入口关系跑通
- 可以尽快验证产品信息架构

代价是：

- 前后端真实实体边界暂时不完全一致
- 首页、Session、导出、标注等页面现在更偏“产品壳层已成型”

### 2. 当前最稳的真实中心仍然是任务 + 资产 + QC + 作业

如果要判断“现在哪一层最可信”，当前最稳的真实中心仍然是：

- 任务
- 文件
- 资产
- 基础 QC
- 处理作业壳层

如果继续推进，应优先围绕这一层继续做后端增强，而不是再继续扩张新的前端概念页。

## 六、建议的下一步演进优先级

### P0：补齐真实 Session

建议优先落地：

- `session` 主表
- Session 列表与详情接口
- 任务与 Session 的真实一对多关系
- 资产、处理作业、QC 报告与 Session 的真实归属

### P1：补齐首页概览专用接口

建议新增首页专用聚合接口，统一提供：

- 资产总数
- 采集批次数
- 成品资产数
- 总时长
- 分布图数据
- 30 天趋势数据

这样可以把首页从“前端聚合 mock 看板”升级为“真实业务概览看板”。

### P1：补齐处理结果真实执行

建议将：

- `processing_job`

从当前的作业骨架，进一步接入真实处理脚本或 worker，至少完成 1 条真实处理链路。

### P2：补齐标注 / 质检规则 / 导出聚合接口

当前这几类页面已经有 UI 骨架，下一步适合补：

- 标注任务接口
- 标注状态与标签接口
- 质检规则接口
- 成品资产聚合接口

## 七、当前结论

截至 `2026-05-22`，MMDP 当前已经具备：

- 真实可运行的后端任务、文件、资产、基础 QC、处理作业接口
- 已重构完成的前端多页面控制台式信息架构
- 已可运行的首页概览页、固定导航、Logo 顶栏、统一页面骨架

但同时仍然存在：

- Session 实体尚未真实落地
- 首页统计和趋势主要由前端聚合 / mock 支撑
- 标注、质检规则、导出等页面仍偏产品壳层
- 处理作业的算法执行仍未完全真实化

一句话最终判断：

> 当前项目已经进入“可持续演进的多模态数据平台骨架阶段”，前端产品结构已经基本成型，后端真实业务骨架已具雏形，下一步最关键的是把 Session、首页聚合和真实处理执行补齐。
